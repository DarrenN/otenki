#+title: otenki
#+subtitle: weather at a glance
#+author: otenki contributors
#+language: en

TUI for viewing weather across multiple locations.

** ✨ AI Disclosure

Large portions of the codebase are AI-assisted. For more information checkout =CLAUDE.md= and =/docs/plans=.

** Features

- Multi-location card grid displaying current conditions side by side
- Responsive grid layout that adapts to terminal width
- Auto-refresh on a configurable timer (default: every 10 minutes)
- 12-hour hourly forecast with temperature per location
- =--json= mode for machine-readable output suitable for scripts and agents
- Metric and imperial units — user-configurable, CLI-overridable
- Inline error cards when a location fetch fails
- Functional Core, Imperative Shell architecture

* Installation

** Clone with submodules

#+begin_src sh
git clone --recurse-submodules https://github.com/youruser/otenki.git
cd otenki
#+end_src

** Build

#+begin_src sh
make build
#+end_src

This compiles a standalone SBCL image and writes it to =bin/otenki=.

** Install

#+begin_src sh
make install
#+end_src

Copies =bin/otenki= to =~/bin/otenki=.  Ensure =~/bin= is on your =PATH=.

** Prerequisites

- [[https://www.sbcl.org/][SBCL]] (Steel Bank Common Lisp)
- [[https://www.quicklisp.org/][Quicklisp]]
- An OpenWeatherMap API key (free tier works fine — see [[#configuration][Configuration]])

** Configuration
  :PROPERTIES:
  :CUSTOM_ID: configuration
  :END:

*** API Key

otenki reads your OpenWeatherMap API key from the environment variable
=OPENWEATHER_API_KEY=.

Get a free key at [[https://openweathermap.org/api][openweathermap.org/api]], then export it in your shell
profile:

#+begin_src sh
export OPENWEATHER_API_KEY=your_key_here
#+end_src

*** Config File

Create =~/.config/otenki/config.lisp= with a plist:

#+begin_src lisp
(:units :metric
 :refresh-interval 600
 :locations ("Tokyo" "London" "New York"))
#+end_src

| Key                | Type            | Default    | Description                        |
|--------------------+-----------------+------------+------------------------------------|
| =:units=           | =:metric= / =:imperial= | =:metric=  | Display unit system              |
| =:refresh-interval= | integer (seconds) | =600=    | Auto-refresh cadence               |
| =:locations=       | list of strings | none       | Locations shown on startup         |

Resolution order: CLI args > config file > built-in defaults.

** Usage

#+begin_src sh
otenki                          # launch TUI with config locations
otenki Tokyo London             # launch TUI for specific locations
otenki --json Tokyo             # one-shot JSON output to stdout
otenki --units imperial         # override units for this session
otenki -h                       # show help
#+end_src

*** Keybindings

| Key | Action        |
|-----+---------------|
| =q= | Quit          |
| =r= | Force refresh |

*** JSON Output

=--json= mode skips the TUI and prints a JSON array to stdout, one object
per location.  Useful in scripts and AI agent pipelines:

#+begin_src sh
otenki --json Tokyo | jq '.[0].condition'
#+end_src

Each object contains:

| Field           | Type    | Description                             |
|-----------------+---------+-----------------------------------------|
| =location=      | string  | Location name                           |
| =lat=           | number  | Latitude                                |
| =lon=           | number  | Longitude                               |
| =temp_c=        | number  | Current temperature (°C, 1 decimal)     |
| =feels_like_c=  | number  | Feels-like temperature (°C, 1 decimal)  |
| =humidity=      | integer | Humidity percentage                     |
| =wind_speed_ms= | number  | Wind speed (m/s)                        |
| =wind_dir=      | integer | Wind direction (degrees)                |
| =condition_id=  | integer | OWM condition code                      |
| =condition=     | string  | Human-readable condition                |
| =hourly=        | array   | Up to 12 hourly forecast entries        |

Each =hourly= entry contains:

| Field          | Type    | Description                         |
|----------------+---------+-------------------------------------|
| =hour=         | integer | Hour of day (0–23, local time)      |
| =temp_c=       | number  | Temperature (°C, 1 decimal)         |
| =condition_id= | integer | OWM condition code                  |
| =pop=          | number  | Probability of precipitation (0–1)  |

** Development

*** Quickstart

#+begin_src sh
make deps     # initialise git submodules (vendors)
make test     # run FiveAM test suite
make repl     # start an interactive SBCL REPL with the system loaded
make build    # compile the otenki executable
make install  # copy to ~/bin/otenki
make clean    # remove build artefacts
#+end_src

*** Running tests

#+begin_src sh
make test
#+end_src

Tests use [[https://github.com/lispci/fiveam][FiveAM]] and never require a live API key or network access — all
HTTP responses are provided as canned JSON fixtures under =t/fixtures/=.

*** Interactive REPL

#+begin_src sh
make repl
#+end_src

Inside the REPL:

#+begin_src lisp
(asdf:load-system :otenki)
(otenki.main:main)
#+end_src

*** Running a single test suite

#+begin_src lisp
(ql:quickload :otenki/tests)
(5am:run! 'otenki.tests:all-tests)
#+end_src

*** Architecture

otenki follows the Functional Core, Imperative Shell pattern using
[[https://github.com/atgreen/cl-tuition][cl-tuition]]'s TEA (The Elm Architecture) loop.

#+begin_example
┌─────────────────────────────────────────────────────┐
│                    main.lisp                        │
│              CLI dispatch, entry point              │
├─────────────────────────────────────────────────────┤
│                    app.lisp                         │
│           TEA wiring: init, update, view            │
├──────────┬──────────┬──────────┬─────────┬──────────┤
│model.lisp│config.lisp│ api.lisp│view.lisp│json.lisp │
│ structs  │ parsing  │ OWM API  │rendering│  JSON    │
│ & units  │ & merge  │ wrapper  │ (pure)  │  output  │
└──────────┴──────────┴──────────┴─────────┴──────────┘
#+end_example

- *Pure functions:* model, api (parsing), view, json — no side effects
- *Imperative shell:* app (TEA wiring), main (CLI), config (file I/O)
- *Messages:* =weather-received-msg=, =weather-error-msg=, =refresh-msg=
- *Commands:* fetch per location, batch fetch, auto-refresh tick
